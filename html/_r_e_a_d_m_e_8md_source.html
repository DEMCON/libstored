1 # libstored - Store for Embedded Debugger
2 
3 This is a generator for a store. A store is a collection of objects, like
4 variables in a `struct`. These objects can be read and written, using common
5 get/set methods. However, the store has a few interesting properties:
6 
7 - Objects are defined by a simple language. The generator produces an
8  application-specific store implementation, and all files needed for your
9  specific project.
10 - Objects can have a piece of memory as backing (just like a normal variable in
11  a `struct`), but can also have custom callbacks on every get and set. This
12  allows all kinds of side effects, even though the interface of the object is
13  the same.
14 - Objects are accessible using a C++ interface, but also via name lookup by
15  string. The generator generates a compact name parser, such that names,
16  types, and sizes can be queried dynamically.
17 - The Embedded Debugger can be attached to a store, which exposes the full
18  store to any external interface. The protocol's application layer is
19  implemented and ready to use. As an application developer, you only have to
20  implement the hardware-specifics, like the transport layer of the Embedded
21  Debugger protocol. You can also easily extend the default set of Embedded
22  Debugger commands by adding another capability in a subclass of
23  stored::Debugger.
24 - There are sufficient hooks by the store to implement an application-specific
25  synchronization method, other than the Embedded Debugger.
26 - All code is normal C++, there are no platform-dependent constructs used.
27  Therefore, all platforms are supported: Windows/Linux/Mac/bare
28  metal (newlib), x86/ARM, gcc/clang/MSVC).
29 
30 Have a look in the `examples` directory for further in-depth reading.

44 ## <a name="build"></a>How to build
45 
46 Make sure to update the submodules after checkout:
47 
48  git submodule init
49  git submodule update
50 
51 Run `scripts/bootstrap` once to install all build dependencies.
52 Then run `scripts/build` to build the project. This does effectively:
53 
54  mkdir build
55  cd build
56  cmake ..
57  cmake --build .
58 
59 By default, all examples are built. For example, notice that sources are
60 generated under `examples/1_hello`, while the example itself is built in the
61 `build` directory. The documentation can be viewed at
62 `doxygen/html/index.html`.
63 
64 To run all tests:
65 
66  cmake --build . -- test

68 ### <a name="integrate"></a>How to integrate in your build
69 
70 Building libstored on itself is not too interesting, it is about how it can
71 generate stuff for you. This is how to integrate it in your project:
72 
73 - Add libstored to your source repository, for example as a submodule.
74 - Run `scripts/bootstrap` in the libstored directory once to install all
75  dependencies.
76 - Include libstored to your cmake project. For example:
77 
78  set(LIBSTORED_EXAMPLES OFF CACHE BOOL "Disable libstored examples" FORCE)
79  set(LIBSTORED_TESTS OFF CACHE BOOL "Disable libstored tests" FORCE)
80  set(LIBSTORED_DOCUMENTATION OFF CACHE BOOL "Disable libstored documentation" FORCE)
81  add_subdirectory(libstored)
82 
83 - Optional: install `scripts/st.vim` in `$HOME/.vim/syntax` to have proper
84  syntax highlighting in vim.
85 - Add some store definition file to your project, let's say `MyStore.st`.
86  Assume you have a target `app` (which can be any type of cmake target), which
87  is going to use `MyStore.st`, generate all required files. This will generate
88  the sources in the `libstored` subdirectory of the current source directory,
89  a library named `app-libstored`, and set all the dependencies right.
90 
91  add_application(app main.cpp)
92  libstored_generate(app FumoStore.st)
93 
94 - Now, build your `app`. The generated libstored library is automatically
95  built.
96 
97 Check out the examples of libstored, which are all independent applications
98 with their own generated store.

117 ## <a name="debugging"></a>Debugging example
118 
119 To get a grasp how debugging feels like, try the following.
120 
121 - Build the examples, as discussed above.
122 - If you use Windows, execute `scripts/env.cmd` to set your environment
123  properly. In the instructions below, use `python` instead of `python3`.
124 - Run your favorite `lognplot` instance, e.g., by running `python3 -m lognplot`.
125 - Run `examples/zmqserver/zmqserver`. This starts an application with a store
126  with all kinds of object types, and provides a ZMQ server interface for
127  debugging.
128 - Run `python3 client/gui_client.py -l`. This GUI connects to both the
129  `zmqserver` application via ZMQ, and to the `lognplot` instance.
130 - The GUI window will pop up and show the objects of the `zmqserver` example.
131  If polling is enabled of one of the objects, the values are forwarded to
132  `lognplot`.

134 ## <a name="commands"></a>Embedded Debugger commands
135 
136 A store can be queried using the following commands. These can be extended by
137 an application.
138 
139 Requests always start with a ASCII characters are command.
140 Every request gets a response. Either with actual data, or with ack `!` or nack `?`.
141 Requests are processed in order.
142 
143 This is the OSI application layer of the protocol stack.
144 
145 ### Capabilities
146 
147 Request: `?`
148 
149  ?
150 
151 Response: a list of command characters.
152 
153  ?rwe
154 
155 ### Echo
156 
157 Request: `e` <any data>
158 
159  eHello World
160 
161 Response: <the same data>
162 
163  Hello World
164 
165 ### Read
166 
167 Request: `r` <name of object>
168 
169 The name of the object may be abbreviated, as long as it is unambiguous.
170 
171  r/bla/asdf
172 
173 Response: <ASCII hex value of object>
174 
175 For values with fixed length (int, float), the byte order is big/network
176 endian. For ints, the initial zeros can be omitted. For other data, all bytes
177 are encoded.
178 
179  123abc
180 
181 ### Write
182 
183 Request: `w` <value in ASCII hex> <name of object>
184 
185 See Read for details about the hex value and object name.
186 
187  w10/b/a
188 
189 Response: `!` | `?`
190 
191  !
192 
193 ### List
194 
195 Requests a full list of all objects of all registered stores to the current Embedded Debugger.
196 
197 Request: `l`
198 
199  l
200 
201 Response: ( <type byte in hex> <length in hex> <name of object> `\n` ) * | `?`
202 
203  3b4/b/i8
204  201/b/b
205 
206 ### Alias
207 
208 Assigns a character to a object path. An alias can be everywhere where an
209 object path is expected. Creating aliases skips parsing the object path
210 repeatedly, so makes debugging more efficient. If no object is specified, the
211 alias is removed. The number of aliases may be limited. If the limit is hit,
212 the response will be `?`.
213 The alias name can be any char in the range 0x20 ` ` - 0x7e `~`, except for 0x2f `/`.
214 
215 Request: `a` <char> ( <name of object> ) ?
216 
217  a0/bla/a
218 
219 Response: `!` | `?`
220 
221  !
222 
223 ### Macro
224 
225 Saves a sequence of commands and assigns a name to it. The macro name can be
226 any char in the range 0x20 ` ` - 0x7e `~`. In case of a name clash with an
227 existing non-macro command, the command is executed; the macro cannot hide or
228 replace the command. The separator can be any char, as long as it is not used
229 within a command of the macro definition. Without the definition after the
230 macro name, the macro is removed. The system may be limited in total definition
231 length. The macro string is reinterpreted every time it is invoked. The
232 responses of the commands are merged into one response frame, without
233 separators.
234 
235 Request: `m` <char> ( <separator> <command> ) *
236 
237  mZ r/bla/a e; r/bla/z
238 
239 Response: `!` | `?`
240 
241  !
242 
243 ### Identification
244 
245 Returns a fixed string that identifies the application.
246 
247 Request: `i`
248 
249  i
250 
251 Response: `?` | <UTF-8 encoded application name>
252 
253  libstored
254 
255 ### Version
256 
257 Returns a list of versions.
258 
259 Request: `v`
260 
261  v
262 
263 Response: `?` | <protocol version> ( ` ` <application-specific version> ) *
264 
265  2 r243+trunk beta
266 
267 ### Read memory
268 
269 Read a memory via a pointer instead of the store. Returns the number of
270 requested bytes. If no length is specified, a word is returned.
271 
272 Request: `R` <pointer in hex> ( ` ` <length> ) ?
273 
274  R1ffefff7cc 4
275 
276 Response: `?` | <bytes in hex>
277 
278  efbe0000
279 
280 ### Write memory
281 
282 Write a memory via a pointer instead of the store.
283 
284 Request: `W` <pointer in hex> ` ` <bytes in hex>
285 
286  W1ffefff7cc 0123
287 
288 Response: `?` | `!`
289 
290  !
291 
292 ### Streams
293 
294 Read all available data from a stream. Streams are application-defined
295 sequences of bytes, like stdout and stderr. They may contain binary data.
296 There are an arbitrary number of streams, with an arbitrary single-char name.
297 
298 To list all streams with data:
299 
300 Request: `s`
301 
302 To request all data from a stream, where the optional suffix is appended to the response:
303 
304 Request: `s` <char> <suffix> ?
305 
306  sA/
307 
308 Response: `?` | <data> <suffix>
309 
310  Hello World!!1/
311 
312 Once data has been read from the stream, it is removed. The next call will
313 return new data. If a stream was never used, `?` is returned. If it was used,
314 but it is empty now, the stream char does not show up in the `s` call, but does
315 respond with the suffix. If no suffix was provided, and there is no data, the
316 response is empty.
317 
318 The number of streams and the maximum buffer size of a stream may be limited.

322 ## <a name="protocol"></a>Protocol stack
323 
324 Every embedded device is different, so the required protocol layers are too.
325 What is common, is the Application layer, but as the Transport and Physical
326 layer are often different, the layers in between are often different too.
327 To provide a common Embedded Debugger interface, the client (e.g., GUI, CLI,
328 python scripts), we standardize on ZeroMQ REQ/REP over TCP.
329 
330 Not every device supports ZeroMQ, or even TCP. For this, several bridges are
331 required. Different configurations may be possible:
332 
333 - In case of a Linux/Windows application: embed ZeroMQ server into the
334  application, such that the application binds to a REP socket. A client can
335  connect to the application directly.
336 - Terminal application with only stdin/stdout: use escape sequences in the
337  stdin/stdout stream. `client/stdio_wrapper.py` is provided to inject/extract
338  these messages from those streams and prove a ZeroMQ interface.
339 - Application over CAN: like a `client/stdio_wrapper.py`, a CAN extractor to
340  ZeroMQ bridge is required.
341 
342 Then, the client can be connected to the ZeroMQ interface. The following
343 clients are provided:
344 
345 - `client/ed2.ZmqClient`: a python class that allows easy access to all objects
346  of the connected store. This is the basis of the clients below.
347 - `client/cli_client.py`: a command line tool that lets you directly enter the
348  protocol messages as defined above.
349 - `client/gui_client.py`: a simple GUI that shows the list of objects and lets
350  you manipulate the values. The GUI has support to send samples to `lognplot`.
351 
352 Test it using the `terminal` example, started using the
353 `client/stdio_wrapper.py`. Then connect one of the clients above to it.
354 
355 ### Application layer
356 
357 See above. See stored::Debugger.
358 
359 ### Presentation layer
360 
361 For terminal or UART: In case of binary data, escape all bytes < 0x20 as
362 follows: the sequence `DEL` (0x7f) removes the 3 MSb of the successive byte.
363 For example, the sequence `DEL ;` (0x7f 0x3b) decodes as `ESC` (0x1b).
364 To encode `DEL` itself, repeat it.
365 See stored::AsciiEscapeLayer.
366 
367 For CAN/ZeroMQ: nothing required.
368 
369 ### Session layer:
370 
371 For terminal/UART/CAN: no session support, there is only one (implicit) session.
372 
373 For ZeroMQ: use REQ/REP sockets, where the application-layer request and
374 response are exactly one ZeroMQ message. All layers below are managed by ZeroMQ.
375 
376 ### Transport layer
377 
378 For terminal or UART: out-of-band message are captured using `ESC _` (APC) and
379 `ESC \` (ST). A message consists of the bytes in between these sequences.
380 See stored::TerminalLayer.
381 
382 In case of lossly channels (UART/CAN), CRC, message sequence number, and
383 retransmits should be implemented. This depends on the specific transport
384 hardware and embedded device.
385 
386 ### Network layer
387 
388 For terminal/UART/ZeroMQ, nothing has to be done.
389 
390 For CAN: packet fragmentation/reassembling/routing is done here.
391 
392 ### Datalink layer
393 
394 Depends on the device.
395 
396 ### Physical layer
397 
398 Depends on the device.

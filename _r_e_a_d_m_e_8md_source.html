3 # libstored - Store for Embedded Debugger
4 
5 ## TL;DR
6 
7 **What is it?**
8 A generator for a C++ class (store) with your application's variables, and a
9 tool to access it remotely.
10 
11 **When do I need it?**
12 When you want to be able to inspect and modify internal data of a running
13 application.
14 
15 **Does it work on my platform?**
16 Yes.
17 
18 **How do I use it?**
19 Have a look at the [examples](examples).

31 ## <a name="intro"></a>Introduction
32 
33 If you have an embedded system, you probably want to debug it on-target. One
34 of the questions you often have, is what is the value of internal variables of
35 the program, and how can I change them. Debugging using `gdb` is great, but it
36 pauses the application, which also stops control loops, for example.
37 
38 libstored helps you to access internal variables. These variables are part of
39 a _store_. A store is defined in a simple language, and libstored provides a
40 generator to produce a C++ class (as cmake library) with the variables you
41 want. This generated C++ class is very efficient when accessed by the C++
42 application. However, it adds interesting functionality: all variables can be
43 accessed remotely.
44 
45 The (OSI-stack) Application layer of this debugging interface is provided by
46 libstored. Additionally, other layers are available to support lossless and
47 lossy channels, which fit to common UART and CAN interfaces. You have to
48 combine, and possibly add, and configure other (usually hardware-specific)
49 layers of the OSI stack to get the debugging protocol in and out of your
50 system. Although the protocol fits nicely to ZeroMQ, a TCP stream, or `stdio`
51 via terminal, the complexity of integrating this depends on your embedded
52 device. However, once you implemented this data transport, you can access the
53 store, and observe and manipulate it using an Embedded Debugger (PC) client,
54 where libstored provides Python classes, a CLI and GUI interface.
55 
56 Your application can have one store with one debugging interface, but also
57 multiple stores with one debugging interface, or one store with multiple
58 debugging interfaces -- any combination is possible.
59 
60 The store has a few other interesting properties:
61 
62 - Objects can have a piece of memory as backing (just like a normal variable in
63  a `struct`), but can also have custom callbacks on every get and set. This
64  allows all kinds of side effects, even though the interface of the object is
65  the same.
66 - Objects are accessible using a C++ interface, but also via name lookup by
67  string. The generator generates a compact name parser, such that names,
68  types, and sizes can be queried dynamically.
69 - The Embedded Debugger has a standard set of commands, which can be disabled
70  for your specific project. Moreover, you can also easily add custom commands
71  by adding another capability in a subclass of stored::Debugger.
72 - There are sufficient hooks by the store to implement any application-specific
73  synchronization method, other than the Embedded Debugger.
74 - The store and all other libstored classes are not thread-safe.
75  Using threads is troubling anyway, use [fibers](https://github.com/jhrutgers/zth) instead.
76 - All code is normal C++, there are no platform-dependent constructs used.
77  Therefore, all platforms are supported: Windows/Linux/Mac/bare
78  metal (newlib), x86/ARM, gcc/clang/MSVC/armcc).
79 
80 Have a look in the `examples` directory for further in-depth reading.
81 Refer to the [Doxygen documentation](https://demcon.github.io/libstored) for the C++ API.

83 ## <a name="build"></a>How to build
84 
85 Run `scripts/bootstrap` (as Administrator under Windows) once to install all
86 build dependencies. Then run `scripts/build` to build the project. This does
87 effectively:
88 
89  mkdir build
90  cd build
91  cmake ..
92  cmake --build .
93 
94 `scripts/build` takes an optional argument, which allows you to specify the
95 `CMAKE_BUILD_TYPE`. If not specified, Debug is assumed.
96 
97 By default, all examples are built. For example, notice that sources are
98 generated under `examples/1_hello`, while the example itself is built in the
99 `build` directory. The documentation can be viewed at
100 `doxygen/html/index.html`.
101 
102 To run all tests:
103 
104  cmake --build . -- test

106 ### <a name="integrate"></a>How to integrate in your build
107 
108 Building libstored on itself is not too interesting, it is about how it can
109 generate stuff for you. This is how to integrate it in your project:
110 
111 - Add libstored to your source repository, for example as a submodule.
112 - Run `scripts/bootstrap` in the libstored directory once to install all
113  dependencies.
114 - Include libstored to your cmake project. For example:
115 
116  set(LIBSTORED_EXAMPLES OFF CACHE BOOL "Disable libstored examples" FORCE)
117  set(LIBSTORED_TESTS OFF CACHE BOOL "Disable libstored tests" FORCE)
118  set(LIBSTORED_DOCUMENTATION OFF CACHE BOOL "Disable libstored documentation" FORCE)
119  add_subdirectory(libstored)
120 
121 - Optional: install `scripts/st.vim` in `$HOME/.vim/syntax` to have proper
122  syntax highlighting in vim.
123 - Add some store definition file to your project, let's say `MyStore.st`.
124  Assume you have a target `app` (which can be any type of cmake target), which
125  is going to use `MyStore.st`, generate all required files. This will generate
126  the sources in the `libstored` subdirectory of the current source directory,
127  a library named `app-libstored`, and set all the dependencies right.
128 
129  add_application(app main.cpp)
130  libstored_generate(app FumoStore.st)
131 
132 - Now, build your `app`. The generated libstored library is automatically
133  built.
134 
135 Check out the examples of libstored, which are all independent applications
136 with their own generated store.

138 ## <a name="syntax"></a>Syntax example
139 
140 See the [examples](https://demcon.github.io/libstored/examples.html) directory
141 for more explanation. This is just an impression of the syntax.
142 
143  // Comment
144  // Grammar: type:size[array]=initializer long name with any character
145 
146  uint32 some int
147  int8=42 another int, which is initialized
148  (uint64) time (s)
149 
150  {
151  bool=true b
152  double[2] numbers
153  string:16 s
154  } scope
155 
156 The generated store has variables that can be accessed like this:
157 
158  mystore.some_int = 10;
159  int i = mystore.another_int_which_is_initialized;
160  mystore.time_s.get();
161  mystore.scope__b = false;
162  mystore.scope__numbers_0.set(0.1);
163  mystore.scope__numbers_1.set(1.1);
164  mystore.scope__s.set("hello");

166 ## <a name="debugging"></a>Debugging example
167 
168 The host tools to debug your application are written in python, as the `ed2`
169 package, and are located the `client` directory. You can run the example below
170 by running python from the `client` directory, but you can also install the
171 `ed2` package on your system. To do this, execute the `ed2-install` cmake
172 target, such as:
173 
174  cd build
175  make ed2-install
176 
177 This builds a wheel from the `client` directory and installs it locally using
178 `pip`. Now you can just fire up python and do `import ed2`.
179 
180 To get a grasp how debugging feels like, try the following.
181 
182 - Build the examples, as discussed above.
183 - If you use Windows, execute `scripts/env.cmd` to set your environment
184  properly. In the instructions below, use `python` instead of `python3`.
185 - Run your favorite `lognplot` instance, e.g., by running `python3 -m lognplot`.
186 - Run `examples/zmqserver/zmqserver`. This starts an application with a store
187  with all kinds of object types, and provides a ZeroMQ server interface for
188  debugging.
189 - Run `python3 -m ed2.gui -l` within the `client` directory. This GUI connects
190  to both the `zmqserver` application via ZeroMQ, and to the `lognplot` instance.
191 - The GUI window will pop up and show the objects of the `zmqserver` example.
192  If polling is enabled of one of the objects, the values are forwarded to
193  `lognplot`.
194 
195 The structure of this setup is:
196 
197  +---------+ +----------+
198  | ed2.gui | -----> | lognplot |
199  +---------+ +----------+
200  |
201  | ZeroMQ REQ/REP channel
202  |
203  +-----------+
204  | zmqserver |
205  +-----------+
206 
207 ![zmqserver debugging screenshot](examples/zmqserver/zmqserver_screenshot.png)
208 
209 The Embedded Debugger client connects via ZeroMQ.
210 If you application does not have it, you must implement is somehow.
211 The `examples/terminal/terminal` application could be debugged as follows:
212 
213 - Run `python3 -m ed2.wrapper.stdio ../build/examples/terminal/terminal` from
214  the `client` directory. This starts the `terminal` example, and extracts
215  escaped debugger frames from `stdout`, which are forwarded to a ZeroMQ
216  interface.
217 - Connect a client, such as `python3 -m ed2.gui`.
218 
219 The structure of this setup is:
220 
221  +---------+
222  | ed2.gui | terminal interface
223  +---------+ |
224  | |
225  | ZeroMQ REQ/REP channel |
226  | |
227  +-------------------+ |
228  | ed2.wrapper.stdio | -----------------+
229  +-------------------+
230  |
231  | stdin/stdout (mixed terminal interface
232  | with Embedded Debugger messages)
233  |
234  +----------+
235  | terminal |
236  +----------+
237 
238 There are some more ready-to-use clients, and a Python module in the
239 [client](https://github.com/DEMCON/libstored/tree/master/client) directory.

241 ## <a name="protocol"></a>Embedded Debugger protocol
242 
243 Communication with the debugger implementation in the application follows a
244 request-response pattern. A full description of the commands can be found in
245 the [doxygen documentation](https://demcon.github.io/libstored/group__libstored__debugger.html).
246 These commands are implemented in the stored::Debugger class and ready to be
247 used in your application.
248 
249 However, the request/response messages should be wrapped in a OSI-like protocol stack, which is described in more detail in the
250 [documentation too](https://demcon.github.io/libstored/group__libstored__protocol.html)).
251 This stack depends on your application. A few standard protocol layers are
252 available, which allow to build a stack for lossless channels (stdio/TCP/some
253 UART) and lossy channels (some UART/CAN). These stacks are configurable in
254 having auto retransmit on packet loss, CRC-8/16, segmentation, buffering, MTU
255 size, ASCII escaping and encapsulation. See also `examples/7_protocol`.
256 
257 To get a grasp about the protocol, I had a short chat with the `zmqserver`
258 example using the `ed2.cli`. See the transcript below. Lines starting
259 with `>` are requests, entered by me, lines starting with `<` are responses
260 from the application.
261 
262 In the example below, I used the following commands:
263 
264 - `?`: request capabilities of the target
265 - `l`: list object in the store
266 - `i`: return the identification of the target
267 - `r`: read an object
268 - `w`: write an object
269 - `v`: request versions
270 - `a`: define an alias
271 
272 Refer to the documentation for the details about these and other commands.
